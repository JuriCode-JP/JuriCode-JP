## 1. v0.2 仕様設計の確定事項

### 1.1 3 層構造 (Three-layer architecture)

v0.1 が「条単位 chunk」だったのに対し、v0.2 は以下の 3 層 + rollup で構成されます。

| 層 | 内容 | 生成元 | chunk 数 (v0.2.2 時点) |
|---|---|---|---|
| 本則 segment | 本文 / ただし書 / 前段 / 後段 / 柱書 等 | `data/**/*.md` → `tools/parse/v0.2/segment_parser.py` | ~17,967 |
| 各号 (kou) | Article > Paragraph > Item | `cache/laws/{law_id}.xml` → `tools/parse/v0.2/extract_kou_from_xml.py` | ~14,868 |
| 附則 (SupplProvision) | 経過措置 / 施行期日 / 罰則の適用 等 | `cache/laws/{law_id}.xml` → `tools/parse/v0.2/extract_supplproviso_from_xml.py` | ~23,000+ |
| rollup | 各 article の全 segment 統合 chunk | `tools/parse/v0.2/add_rollup_chunks.py` | ~6,948 |

合計: **63,246 chunks** (v0.2.2)。

### 1.2 Segment type 8 種類

`segment_type` フィールドは以下のいずれか:

`honbun` / `tadashi` / `zen_dan` / `kou_dan` / `hashira` / `kou` / `tokusoku` / `junyou` / `rollup`

### 1.3 4-Layer Responsibility Matrix

| Layer | 責務 | 実装 |
|---|---|---|
| L1 前処理 | XML → Markdown、segment 分割、各号抽出 | `tools/parse/v0.2/` |
| L2 メタデータ | `segment_type` / `modality` / `parent_section` / `target_main_articles` 付与 | `segment_parser.py` + `extract_kou_from_xml.py` |
| L3 検索 | dense + BM25 hybrid + `dedup_by_article` | `tools/embed/retrieve.py` |
| L4 prompt | LLM 生成時に metadata を活用 (modality、章 scope、準用) | **未実装 (Phase 2)** |

### 1.4 ファイル単位 = Option A 採用

- **1 条 = 1 .md file** を v0.1 から維持
- parser が **`.chunks.jsonl` を別途生成** (1 article = N chunks)
- 23 軸 × 5 条文の検証で Option A が 17/23 軸優位
- Obsidian Vault 親和性、`CLAUDE.md` の v0.1 規約を破壊しない

### 1.5 SupplProvision (附則) の metadata 拡充

附則 chunk に付与するフィールド:

- `amend_law_num` — 改正法番号 (例: 平成26年法律第69号)
- `amend_era` / `amend_year_gengo` / `amend_year_seireki` — **元号→西暦変換**
- `amend_month` / `amend_day` / `amend_law_num_int`
- `enforcement_date` — 施行日
- `topic` — 経過措置 / 施行期日 / 罰則の適用 / 検討規定 (ArticleCaption + text heuristic で分類)
- `target_main_articles_raw` — 「第X条」等の raw 抽出
- `target_main_article_ids` — `{law_abbrev}-art-{N}` 形式に正規化

### 1.6 Rollup chunk (parent-child design)

各 article について全 segment を統合した chunk を生成 (`segment_type: "rollup"`)。retrieval 時の `dedup_by_article` で親 chunk として機能します。

### 1.7 v0.1 → v0.2 主要変更点まとめ

| 観点 | v0.1 | v0.2 |
|---|---|---|
| chunk 粒度 | 条単位 | 項 + segment 単位 |
| 各号 content | 欠落 (parser bug) | XML 再取得で完全復元 |
| 附則 | 未収録 | metadata enrichment 込みで収録 |
| modality | 未付与 | 文末判定で 8 種 |
| target reference | 未抽出 | raw + 正規化 ID |

### 2.2 Benchmark 数値 (記憶ベース、再測定推奨)

**Municipal (自治体クエリ)**
- v0.1 baseline: R@1 = 65.7%
- v0.2.2: R@1 = **68.6%** (+2.9pt) → 改善達成 ✅

**Lawqa-jp**
- v0.1: R@3 = 68.6%
- v0.2.2: R@3 = 54.7% (-13.9pt) → **regression、未解決** ⚠️
- 原因仮説: chunk 細分化で keyword 分散、BM25 weight 要調整

**6,984 件 (Phase 1 完了時 small-N)**
- Recall@3 = 100% / MRR = 0.875

### 2.3 適用した精度対策

#### A. Hybrid retrieval (dense + BM25)
- dense: `gemini-embedding-001` (3072 dim)
- sparse: BM25 (sudachi 分かち書き)
- α 加重平均で fusion

#### B. dedup_by_article (★重要修正)
- 同一 article から複数 chunk が top にあるとき rollup 優先
- **発見した bug**: dedup が `dense_top_idx` を直接使い、hybrid 結果を無視していた
- 修正: `top_idx_wide` (post-hybrid) で dedup、候補プールを wide に

#### C. 枝番条 regex 修正 (★重要修正)
- 第N条のM (例: 第3条の2) が `PARAGRAPH_HEADING_PATTERN` にマッチせず
- → 4,810 個の **empty `.chunks.jsonl`** が発生していた
- 修正: pattern と body split の両方に `(?:の[零〇一二三四五六七八九十百千万]+)?` を追加

#### D. SupplProvision metadata enrichment
- 元号→西暦変換 → 時系列フィルタ可能
- `target_main_article_ids` → 「附則 X 条 → 本則 Y 条」リンク
- `topic` 分類 → 経過措置 vs 施行期日を区別

#### E. Rollup chunk 導入
- segment 細かさを保ちつつ、article 全文を rollup として保持
- dedup 時の親 chunk として機能

### 2.4 未解決の課題

1. **Lawqa-jp R@3 regression** — chunk 細分化の副作用
   - 対策案: BM25 weight up、rollup chunk priority boost、reranker 導入
2. **bulk-ingest tag bug** — `phase1-tax` が `phase1-police` に書き換えられる事故
   - `git checkout HEAD -- data/` で 2,003 file revert 済、根本原因未修正
3. **L4 prompt layer 未実装** — modality / 章 scope / 準用を LLM 生成時に活用していない

## 3. 開発で踏んだ落とし穴と対策

### 3.1 Path.with_suffix() bug with dotted prefix
- **症状**: `prefix.with_suffix(".npy")` が `v0.2-gemini-17967` のような dotted prefix で誤動作 (`.2-gemini-17967` を suffix 扱い)
- **対策**: `prefix.parent / (prefix.name + ".npy")` を使う。`retrieve.py` / `embed.py` 両方で修正済

### 3.2 dedup_by_article が hybrid 結果を無視
- **症状**: `dense_top_idx` を直接参照、BM25 weight が消失
- **対策**: `top_idx_wide` (post-hybrid) で dedup

### 3.3 枝番条 regex 抜け落ち
- **症状**: 4,810 件の empty `.chunks.jsonl`
- **対策**: PARAGRAPH_HEADING_PATTERN と body split regex 両方に `(?:の[零〇一二三四五六七八九十百千万]+)?`

### 3.4 ファイル corruption (Edit/Write tool 截断)
- **症状**: 大きな差分で末尾截断、NUL byte padding
- **対策**: Write 後は `tail -5` + `python -m py_compile` で必ず検証

### 3.5 cat heredoc append による二重 `if __name__` block
- **症状**: Edit/Write 截断後に `cat >> file << EOF` で復旧すると、既存の `if __name__` block の後ろに同じ block が貼り付けられる。ruff format が parse error で落ちる
- **対策**: 截断したら `cat >> EOF` ではなく、ファイル全体を別 Path に Write し直す

### 3.6 ruff F541 (f-string without placeholder)
- **症状**: `f"some text"` のような placeholder なしの f-string で CI 落ち
- **対策**: コミット前に Cowork 側で `ruff check --fix` を回す

### 3.7 WSL /mnt/c/ ruff format による複数ファイル corruption
- **症状**: WSL から /mnt/c/ のファイルに `ruff format` で 5 ファイル同時 corruption (2026-05-22)
- **対策**: lint/format は **Cowork (Linux サンドボックス) 側で実行**。WSL からは触らない

### 3.8 bulk-ingest tag bug
- **症状**: regenerate された v0.1 .md の tag が一律 `phase1-police` に書き換え
- **対策**: `git checkout HEAD -- data/` で revert。根本原因は Phase 2 で対応

### 3.9 SSL retry on WSL2/Gemini
- **症状**: Gemini embedding 大量バッチ時に SSL handshake 失敗
- **対策**: tenacity exponential backoff、max 5 retries

### 3.10 Article rollup chunks vs segment 細分化の trade-off
- **症状**: 細分化で municipal は上がるが、lawqa-jp は下がる
- **対策**: rollup を retrieval 候補に含める。完全解決には reranker 必要
