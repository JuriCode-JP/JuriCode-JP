# 段階戦略 — Phase 1〜3

JuriCode-JPは普遍ミッション(日本の法令をAI時代に最適化する)を掲げつつ、**段階的フォーカス**で社会課題解決の実効性とプロジェクト持続性を両立させる。

---

## ミッション

> **北極星(最終ビジョン)**: **すべての公務員が安心して使える、検証済みの公共法令インフラ。** 公務員の法令実務を、正確で出典の辿れる ground truth で支える。「安心して使える」は具体的な仕様 — 正確性(本文 == e-Gov 原典・round-trip 検証)/ 出典・追跡可能性(全記述が e-Gov/判例/通達に辿れる)/ ハルシネーションしない(賢い LLM でなく検証済みデータが ground truth)/ 中立(解釈を押し付けず判断は人間に委ねる)/ 責任の所在(人間の判断を支え置き換えない)。**ミッションと堀(検証済み canonical データ)は同じものを指す。** ゆえに JuriCode-JP はアプリ層ではなく、それらが信頼して食べるデータ層に徹する(→ `differentiation.md`)。

四本柱(警察・自治体・法律実務家・税理士)は入口、北極星は全公務員・全領域。各領域で「安心」を成立させるのが下記「深さの軸」(検証・通達・改正対応)。

🤖 AI/LLM向け法令データの提供
⚖️ 判例リンクによる立体的な法令理解
🌏 英訳併記による国際的アクセシビリティ
🌱 オープンソースによる持続可能な発展

---

## Phase 1: 国内四本柱の実用基盤 (2026-2027)

> **2026-05-20 拡張**: 当初は「警察関連法令」のみを Phase 1 と定義していたが、
> 実装が想定より速く進んだことを受け、Phase 1 のスコープを**国内 4 つのターゲット層**
> を支える法令群へ拡張した(警察 + 自治体 + 法律実務家 + 税理士)。

### 四本柱とカバー法令

| 柱 | ターゲット | 投入済み・予定法令 |
|---|---|---|
| 1. 警察 | 警察庁・警視庁・各都道府県警 | 刑法、刑事訴訟法、警察法、警察官職務執行法 (Phase 1 投入済 1,118 条) |
| 2. 自治体 | 都道府県・市区町村職員 | 地方自治法、行政手続法、行政不服審査法 (Phase 1 投入済 651 条) |
| 3. 法律実務家 | 中小法律事務所・行政書士・司法書士・社労士 | 民法 (Phase 1 投入済 1,165 条)、商法・会社法 (Phase 2 で拡張) |
| 4. 税理士 (2026-05-20 追加) | 税理士・企業税務担当 | 国税通則法、法人税法、所得税法、消費税法 (Phase 1 投入予定 ~660 条) |

### 解決する社会課題

- 自治体の条例整備・住民問合せ・議会答弁の根拠提示の効率化
- 中小法律事務所の業務効率化(商用 DB 高額・中小には届かない)
- 税理士・税務担当の通達照合・申告書類根拠確認の効率化
- 警察人手不足への直接的アプローチ
- LegalRAG 精度向上のための構造化基盤

### 想定応用
- 警察調書作成支援AI(他者開発含む)
- 自治体 DX 補助ツール / 議会答弁支援
- 中小法律事務所向け参照基盤
- 税務 AI / 通達連携アプリ

## 深さの軸: 構造化の深掘り (v0.3〜) — Phase と直交する第二の軸

Phase 1〜3 は「**広さ**」(どの法令をカバーするか) の軸である。これと**直交する第二の軸が「深さ」**(各法令をどこまで実用レベルに構造化するか) であり、v0.3 以降の中心テーマとなっている。法令本文を入れただけでは実務 (とくに税務・警察) では足りず、通達・施行令・別表・判例・英訳まで縦に掘って初めて使える。

深さの軸の構成要素 (全法令に共通):
- **法令本文** (e-Gov・round-trip 検証済) → **施行令・施行規則** → **別表・税率表の TableStruct** (FU-515) → **基本通達** (Pydantic DirectiveChunk・FU-514) → **質疑応答事例 / タックスアンサー** → **判例リンク** → **英訳併記** (公定訳 JLT)。
- 横断インフラ: format-spec v0.2 (segment + 項単位 chunk)、retrieval (dense embedding baseline)、検索 UI、源内接続。実装は内部 business 計画の柱1〜5 で管理。

### 税務の深掘りラダー (税理士・税務向け下流アプリ直結・最優先の縦堀り)

税理士実務に必要な情報体系は法令だけでは不足し、下表の各層が要る。`%` は概算 (2026-06-24 時点)。

| 層 | 状態 | 概算 |
|---|---|---|
| 税法本文 (国税通則・法人税・所得税・消費税・相続税・地方税) | 投入済 | ~100% |
| 各施行令・施行規則 | 投入済 (二重管理も解消済 FU-516) | ~100% |
| 基本通達 | 消費税=第1章完了→残18章 FU-519 / 法人税=役員給与節のみ / 他 (所得税・相続税・財産評価・国税通則) 未着手 | ~15% |
| タックスアンサー / 質疑応答事例 | 法人税の少数のみ | ~5% |
| 租税特別措置法 (本文 + 措置法通達) | 未着手 (Phase 3 に明記) | 0% |

→ 「税法本文」を分母にすればほぼ完成だが、「税理士実務の情報体系」を分母にすると **おおむね 35〜45%**。残りの大物は順に: 消費税通達の完成 (FU-519) → 法人税通達の全体化 → 所得税・相続税・財産評価の各通達 → タックスアンサー拡充 → 租税特別措置法。これが「広さより深掘り」の本丸であり、下流の実務ツール（行政・税務）の retrieval 実用価値に直接効く。

### 改正履歴レイヤー(将来フェーズ・資源待ちの北極星)

北極星(全公務員の安心)に直結する深さの一軸が**改正履歴の構造化**。行政実務では「この条文は改正前後でどう変わったか/旧条文は何か/なぜ変えたか」を要し、これは改正版を追える canonical データがないと答えられない。検索向けの法令ツールは改正附則を落とすものが多く(`differentiation.md`)、ここは大きな差別化になる。

- **対象**: e-Gov の改正履歴が遡れる **2016 年以降**(一次データの下限)を point-in-time でデータ化。
- **現状**: format は既に改正を一部予約済(frontmatter の `version_date` / `amendments[]`、安定 `article_id`)。**先送りなのは (a) 過去版の populate と (b) 版間を横断する point-in-time/差分グラフの addressing**。「スキーマは今予約、ロジックは先送り」の規律どおり、入れる箱は概ね設計済。
- **連携**: 版間の差分・改正グラフは **gitlaw-jp(法令の git 管理)領域**=自前再発明でなく連携/統合が筋。
- **位置づけ**: 専用の資金が要る将来の大型フェーズ。現状は**スキーマ予約と北極星の言語化**まで。当面の資源は現在の深掘り(税務通達)に集中する(ROI 優先)。

## 二段構え戦略の意義

JuriCode-JPは**対外メッセージ**(普遍ミッション)と**内部実行**(警察フォーカス)を分けて運用する。

| 観点 | 対外 | 内部 |
|---|---|---|
| メッセージ | 日本の法令をAI時代に最適化 | Phase 1で警察関連を深掘り |
| 想定読者 | 法律家、企業法務、市民、国際社会 | プロジェクトメンバー、Tokyo Award審査員 |
| 目的 | 排除しない、将来展開を閉ざさない | 社会課題への直接性、応募ストーリーの明確化 |

これにより、Phase 1着手中も Phase 2/3 のステークホルダーからの関心を維持できる。

## 危機シナリオと対応

| リスク | 対応 |
|---|---|
| 大手LegalTech(企業)に類似プロジェクトを先行公開される | OSS x オープンデータの優位性を維持、コミュニティ依存度を上げる |
| e-Gov API仕様の大幅変更 | `tools/fetch-egov/` の抽象化、複数バージョン対応 |
| 判例情報のライセンス制約強化 | citation・概要は引用範囲、リンクのみで本文非掲載に徹する |
| Phase 1の警察協力が得られない | 民間活用(教育・研究・記者用)で価値を立証してから接近 |

## Out of Scope(明示的に扱わないこと)

- 個別事案の法律相談・判定
- 弁護士業務・司法書士業務の代替
- 法令の改正提案(立法府の領域)
- 政治的見解の表明
- 特定政党・特定政策への接近

JuriCode-JPは**インフラ**としての中立性を保つ。
